# 功能页配图资源

## 概述

根据功能页文案的含义制作可上线使用的页面配图。除非用户明确说明现有图片是参考图，否则把页面中的现有图片视为临时占位图。

实际生成图片时使用 `imagegen` skill。本 skill 负责定义文案提取、资产规划、命名、prompt、后处理和校验规则。

## 输出语言

默认用中文进行面向用户的沟通，包括资产规划、区块映射、进度更新、保存路径、prompt 摘要、校验结果和限制说明。文件名、slug、URL、命令、代码、JSON 和文件路径保持原样。

图像生成 prompt 可以使用英文，因为英文通常更利于图像模型稳定执行。如果需要向用户汇报 prompt，默认给中文摘要，不要贴很长的英文 prompt，除非用户明确要求查看完整 prompt。

## AIGAZOU 默认资产模式

当用户提供 AIGAZOU 功能页 URL，例如 `https://aigazou.net/en/features/seedream-4-5`，且没有指定其他要求时，默认生成以下资产：

| 页面区块 | 数量 | 尺寸 | 文件名 |
| --- | ---: | --- | --- |
| 四张竖向展示图下方的功能文案区 | 3 | `1200x800` 横图 | `<slug>-6.webp`, `<slug>-7.webp`, `<slug>-8.webp` |
| "Ideal Use Cases" 卡片区 | 3 | `800x1200` 竖图 | `<slug>-9.webp`, `<slug>-10.webp`, `<slug>-11.webp` |

如果页面有更多已确认区块，继续按顺序编号。如果用户指定起始编号，使用用户给出的编号而不是默认编号。

## 默认视觉方向

默认采用大品牌功能页式构图，参考 Fotor 的 Nano Banana Pro 页面和 BeautyPlus 的 Image Enhancer 页面所体现的构图原则。不要复制它们的具体图片、布局、品牌元素或 UI，只学习底层视觉方法：

- 优先使用一个占主导地位的主视觉，而不是工作流拼贴。主体或主图卡应占画面约 60-75%。
- 每张图限制为 1 个主要视觉模块，最多再加 2 个辅助模块，例如一个细节放大镜、一个前后对比、或一张次级卡片。
- 背景要安静、有氛围：渐变舞台、柔和发光、轻微景深或干净影棚背景。避免堆满桌面道具、大量缩略图、密集 UI 面板和拥挤分镜。
- AI 模型 / 图像生成器页面优先靠近高级深色渐变展示风：深蓝、墨绿、紫色背景，大尺寸圆角图片卡，细描边，柔和阴影，克制的发光点缀。
- 图像增强 / 修图工具页面优先靠近明亮干净的前后对比风：大照片结果，简单 split comparison 或圆形细节放大，白色 / 浅灰 / 柔和粉蓝辅助色，直接展示效果。
- 多展示结果质量，少展示操作流程。UI 元素只作为抽象辅助形状使用；除非页面明确强调界面工作流，否则不要让 UI 成为主角。
- 如果 prompt 显得很忙，删减元素，直到图片能一眼表达一个核心利益点。

## 参考页使用规则

当用户同时提供目标页和竞品参考页时，按下面方式处理：

- 目标页决定“生成什么”：页面文案、功能卖点、use case 顺序、资产数量、文件名和 slug 都以目标页为准。
- 参考页决定“怎么表达”：提取构图方法、大图展示方式、留白比例、主视觉占比、前后对比方式、卡片层级、光影氛围和整体视觉节奏，并抽象成原创配图规则。
- 参考页不是素材库。不要复用其中的具体图片、人物、产品、logo、品牌色、可读文案、页面截图、真实 UI 或专有视觉元素。
- 不要把参考页图片当作图像生成的直接参考图，除非用户明确提供图片并说明可作为参考素材。
- 如果参考页和目标页品类一致，例如同为 AI filter、AI enhancer、AI avatar、AI background remover，优先学习参考页的图片区块结构。
- 如果参考页和目标页品类不一致，只抽象学习通用构图原则，不照搬场景。
- 汇报时用中文说明“从目标页提取的内容”和“从参考页学习的视觉规则”，让用户能确认方向。

推荐用户按这个格式调用：

```text
调用 feature-page-image-assets skill 进行图片生成
目标页：https://aigazou.net/en/features/xxx
参考页：https://competitor.com/ai-filter
参考他们配图的构图和大图展示方式。
配图整体要大气、少元素、主视觉占比高。
```

## 工作流

1. **收集来源**
   - 需要 URL、slug、用户粘贴的页面文案，或文字可读的截图。
   - 如果用户提供 URL，从最后一个路径段提取 slug。
   - 如果用户提供“目标页”，使用目标页提取文案和 slug。
   - 如果用户提供“参考页”，抓取或浏览参考页，归纳其配图区块的构图方法和视觉规则。
   - 使用 `scripts/extract_page_copy.py <url>` 获取干净的页面文案草稿和候选区块。如果动态渲染导致内容不可见，使用浏览器 / DevTools，或请用户提供页面文案。

2. **识别由文案支撑的配图**
   - 读页面文案，不要按当前图片来做图。
   - 忽略临时图片、alt 文本、CDN 图片文件名和现有图片构图，除非用户明确要求匹配它们。
   - AIGAZOU 默认模式下的映射：
     - `-6`：功能总览，通常是统一模型、文本生成图片或图像编辑工作流。
     - `-7`：第一个详细功能卡片 / 功能行。
     - `-8`：第二个详细功能卡片 / 功能行。
     - `-9` 起：按页面顺序对应 use-case 卡片。

3. **规划 prompt**
   - 每个资产单独写一个 prompt。不要在一个 imagegen prompt 里要求生成多张不同资产。
   - 用视觉表达文案中的用户收益。优先使用具体场景、结果展示或视觉隐喻，不要做成页面截图。
   - 如果有参考页，把参考页抽象成构图约束写进 prompt，例如“一个主视觉、大图卡、圆形细节放大、充足留白、少元素”，不要写入参考页品牌名、logo、可读文案或具体图片内容。
   - 保持元素简单、克制、宽松。优先干净、高级、有氛围的构图，避免复杂拼贴和过多细节。
   - 每个 prompt 都要执行大品牌构图约束：一个主视觉、最多 3 个视觉模块、充足留白、不做密集工作流拼贴、不散落缩略图、不做道具堆满的桌面场景，除非文案确实需要。
   - 功能 / 详情区块要把卖点转化为一个大结果展示，而不是把每一步流程都画出来。
   - use-case 卡片每张只表达一个强场景或最终交付物。不要把多个无关例子塞进同一张图。
   - 不要包含可读文字、logo、水印、品牌标识、假 UI 文本或复制网页截图。
   - 不要把旧图 / 临时图当成风格、构图或主体参考。
   - SaaS / 产品页要保持精致、实用、可上线，而不是纯装饰或抽象图。
   - 如果页面有明显品牌色，可以轻度使用；不要让每张图都变成单一色系。
   - 为常见功能 / 用例撰写 prompt 时，读取 `references/prompt-patterns.md`。

4. **生成**
   - 默认使用 `imagegen` skill 的内置 `image_gen` 路径，除非用户明确要求其他图片 API / CLI 流程。
   - 每个不同资产单独调用一次 `image_gen`。
   - 如果生成图片保存在 `$CODEX_HOME/generated_images` 下，把选中的结果复制到项目 / output 目录，并保留原始文件。

5. **后处理**
   - 最终资产转换为 `.webp`。
   - 强制精确尺寸：
     - 功能图：`1200x800`
     - 用例图：`800x1200`
   - 只有在不破坏主体时才使用中心裁切。否则手动选择更合适的裁切位置。
   - 用户没有指定输出位置时，默认保存到 `output/feature-page-image-assets/<slug>/`。
   - 除非用户明确要求替换，否则不要覆盖已有资产。

6. **校验**
   - 确认每个文件存在，扩展名为 `.webp`，尺寸完全符合预期。
   - 尽可能打开或检查图片，避免空图、文字伪影、logo、明显肢体问题或构图不匹配。
   - 汇报保存路径、中文 prompt 摘要和任何限制。

## 命名规则

- 使用 URL slug 作为文件名前缀，只在文件系统安全需要时做必要规范化。
- 使用连字符编号：`<slug>-6.webp`，不要用下划线。
- 按页面区块顺序保持编号稳定。除非用户要求，不要重新编号已生成资产。
- 对本地化 URL，保留功能 slug，不要使用 locale 段。

示例：

- URL: `https://aigazou.net/en/features/seedream-4-5`
- Slug: `seedream-4-5`
- 功能图文件：`seedream-4-5-6.webp`, `seedream-4-5-7.webp`, `seedream-4-5-8.webp`
- 用例图文件：`seedream-4-5-9.webp`, `seedream-4-5-10.webp`, `seedream-4-5-11.webp`

## 何时提问

当 URL 和页面文案明确时，做合理假设并继续执行。

只有在以下情况才向用户简短提问：

- URL 无法抓取，且没有可用文案 / 截图；
- 页面区块数量多于或少于默认模式，导致资产数量不明确；
- 用户需要特定目标文件夹、CDN 命名规则，或需要无法从页面判断的特定视觉风格。

## 常用命令

提取页面文案：

```bash
python3 .codex/skills/feature-page-image-assets/scripts/extract_page_copy.py "https://aigazou.net/en/features/seedream-4-5"
```

如果有 ImageMagick，用它校验尺寸：

```bash
identify -format '%f %wx%h\n' output/feature-page-image-assets/seedream-4-5/*.webp
```

如果没有 ImageMagick，使用项目里已有的图片检查工具，例如 `sharp`、`sips`，或一个小型一次性 Node 脚本。
